# ADR-0007 — Visibility & checkpoint strategy

- **Status**: Accepted
- **Date**: 2026-05-18

## Context

The owner needs to know what the team is doing *while it's happening*, not
only via post-hoc audit. Two distinct surfaces:

1. **`team_feed`** — a live, filterable stream of every inter-agent
   message, redacted enough to be human-readable.
2. **Checkpoint digests** — short structured summaries from Team Lead
   at meaningful milestones.

The owner has stated this is a blocker for Iteration 1: without
visibility, they can't tell whether the team is working correctly.

## Decision

### `team_feed` — three-layer pipeline

```
┌─ Agent A ─┐                  ┌─ Agent B ─┐                  ┌─ Owner CLI ──┐
│ publishes │ ── Redis ──▶     │ may       │                  │ ai-team      │
│ AgentMsg  │   Stream         │ consume   │                  │ watch        │
└──────────┘   (canonical)     │ via       │                  └─────┬────────┘
       │                       │ XREAD     │                        │
       ▼                       └───────────┘                        │
   bus.publish()                                                    │
       │                                                            │
       ├───► [redact_for_feed()] ─► Postgres `feed_events`          │
       │                              (persistent history)          │
       │                                                            │
       └───► [redact_for_feed()] ─► Redis Pub/Sub `team_feed` ──────┤
                                       │                            │
                                       └──► FastAPI SSE bridge ─────┘
                                            /api/feed/stream
```

- **Persistence layer** (Postgres `feed_events`): every published
  message is mirrored, in redacted form, for queryable history. Schema
  includes `message_id`, `correlation_id`, `sender`, `recipient`,
  `message_type`, `priority`, `timestamp`, `summary` (short text),
  `redacted_payload` (JSONB).
- **Real-time fan-out** (Redis Pub/Sub channel `team_feed`): low-latency
  broadcast to all live subscribers (CLI watchers, future web UI).
- **Client API** (FastAPI SSE endpoint `GET /api/feed/stream`): bridges
  Redis Pub/Sub to HTTP for the CLI. SSE chosen over WebSocket because
  the feed is one-directional and SSE has native auto-reconnect.

### Redaction

A `core/security/redaction.py` filter applied *before* publish-to-feed
(and only there — audit log keeps the unredacted payload). It:

- Truncates string fields > 2 KB to `…[TRUNCATED]…`.
- Replaces values of keys matching `(?i)(token|secret|password|key|authorization)`
  with `[REDACTED]`.
- Removes raw HTML / large file contents from tool-result payloads
  (replaced with `[BLOB:size=…,sha256=…]`).
- Drops fields in a per-message-type `feed_safelist` rather than
  attempting blanket allow.

### `ai-team watch` CLI

```
ai-team watch                              # all agents, color-coded
ai-team watch --agent qa_engineer          # one agent
ai-team watch --correlation <id>           # one task tree
ai-team watch --since 10m                  # backfill from history
ai-team watch --no-internal                # default: heartbeats hidden
ai-team watch --json                       # machine-readable
ai-team watch --priority P1,P2             # filter
```

Internally, `watch` connects to SSE; on disconnect, auto-reconnects.
Backfill (`--since`) reads `feed_events` first, then streams forward.

### Agent access to the feed

Agents can call `mcp__ai_team_bus__read_team_feed(filters)` to read
recent feed activity. Read access is intentionally rate-limited (max
3 calls per turn) — feed access is for situational awareness, not for
loading bulk history into context (use audit-log summaries for that;
see [ADR-006]).

### Checkpoint digests — Team Lead → Owner

Triggers (Team Lead initiates):

1. **`task_report{status='done'}`** received from any agent.
2. **Manual** — `ai-team status` / `ai-team digest`.
3. **Scheduled** — every `CHECKPOINT_INTERVAL_MIN` (default 30 min) of
   active work.
4. **End of iteration** — automatically before the iteration's owner
   approval.
5. **Before a `request_human_review`** — required pre-context for any
   pending review.
6. **`P1` alert** — immediate, out of schedule.

Format (≤ 300 tokens, structured markdown):

```
🎯 Checkpoint <type> · <ISO timestamp>
Correlation: <id> · Iteration: <N>

✅ Done
- <agent>: <what + artifact link>

🔄 In progress
- <agent>: <what · pct · ETA>

⛔ Blockers
- <description + who is unblocking + how>

⏳ Awaiting your approval
- <pending_review id · 1-line summary>

📋 Next
- <plan to next checkpoint>

💰 Spend: ~$<n>/day (<m>% of monthly quota)
```

Generated by Haiku (low cost), aggregates from `feed_events` +
`tasks` + `pending_reviews` since the previous digest.

Persisted in table `checkpoints` (id, type, triggered_by, payload_md,
correlation_id, timestamp). Owner can `ai-team digest --history`.

Distribution:

- Published as an `AgentMessage{message_type=CHECKPOINT_DIGEST}` on the
  bus → audit log + feed.
- Push notification to CLI watchers (bell character or terminal
  notification).
- Future Iteration 6: web UI panel.

### Architectural invariants

- **No private DMs**: agents communicate exclusively through the bus.
  Any direct inter-process channel is a bug.
- **Feed publish is non-blocking** (best-effort). Audit log write is
  blocking. If feed Redis is down, messages still flow; only live
  visibility degrades.
- **Audit log is the source of truth.** Feed and digests are
  derivatives.

## Consequences

### Positive

- Owner sees everything in real time without polling.
- Three layers cleanly separate: persistence vs. fan-out vs. transport.
- Digest cadence balances "I want to know what's going on" with "stop
  flooding me with sub-task chatter".
- Cost: Haiku-generated digests are cheap (sub-cent per call).

### Negative

- Two streams to monitor in production (Redis Pub/Sub for feed, Redis
  Streams for canonical messages). We mitigate with shared Redis
  instance and unified monitoring.
- Redaction logic must be kept in sync with new payload types; tested
  via contract suite.

## Alternatives considered

- **Postgres `LISTEN`/`NOTIFY` instead of Redis Pub/Sub.** Rejected —
  8 KB payload limit, persistent-connection coupling, harder horizontal
  scale.
- **WebSocket instead of SSE.** Rejected for now — feed is
  unidirectional, SSE is simpler, plain HTTP, auto-reconnect built-in.
  Iteration 6's bidirectional chat may use WS additionally.
- **Push digests via email/Slack.** Considered for Iteration 5;
  out-of-scope for MVP (owner is at the CLI).
